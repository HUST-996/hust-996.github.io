title: Java

一、Java基础

1.接口和抽象类的区别

相似点：

  • 接口和抽象类都不能被实例化；

  • 实现接口或继承抽象类的普通子类都必须实现这些抽象方法。

不同点：

  • 抽象类可以包含普通方法和代码块，接口里只能包含抽象方法，静态方法和默认方法；

  • 抽象类可以有构造方法，而接口没有；

  • 抽象类中的成员变量可以是各种类型的，接口的成员变量只能是public static final类型的并且必须赋值。

2.重载和重写的区别

重载发生在同一个类中，方法名相同，参数列表、返回类型、权限修饰符可以不同。

3.==和equals的区别

==比较基本类型，比较的是值，==比较引用类型，比较的是内存地址。

equals是Object类的方法，本质上与==一样，但有些类重写了equals方法，比如String的equals被重写后，比较的是字符值，另外重写了equals后，也必须重写hashcode()方法。

4.异常处理机制

使用try、catch、finally捕获异常，finally中的代码一定会执行，捕获异常后程序会继续执行。

使用throws声明该方法可能会抛出的异常类型，出现异常后程序终止。

5.HashMap原理

HashMap是使用数组加链表加红黑树(JDK1.8)来实现的。

HashMap中数组的初始大小为16，数组的初始值都为null。

调用put方法时

比如调用 map.put(99, “huang”)，插入一个key为99的元素。这时候需要利用一个哈希函数来确定Entry1的插入位置(index)；

计算出来后，就将这个Entry存入数组下标为(index的位置)。

而当数组该索引处本身就存在Entry，该怎么办呢？

只需插入到该索引处的链表中即可。

JDK1.8版本使用的是尾插法，并且当链表长度为8时自动转换成红黑树，提高查询效率。

6.有关hashCode

1.hashCode的存在主要是用于查找的快捷性，如HashTable，HashMap等，hashCode是用来在散列存储结构中确定对象的存储地址的；

2.如果两个对象相同(即通过equals比较相同)，则两个对象的hashCode一定要相同；

3.如果对象的equals方法被重写，那么对象的hashCode也尽量重写，并且产生hashCode使用的对象，一定要和equals方法中使用的一致，否则就会违反上面提到的第二点。

4.两个对象的hashCode相同，并不一定表示两个对象就相同，也就是不一定适用于equals，只能说明这俩对象在散列存储结构中，存放在同一个篮子里。

ArrayList扩容机制

  • 当使用无参构造时，初始创建一个长度为0的空数组。

  • 当使用有参构造时，初始创建一个参数长度的数组。

  • 当调用add方法后，容量超过当前数组容量就触发扩容机制，新容量为旧容量的1.5倍（向下取整）。

  • 当调用addAll方法时，如果没有触发扩容机制就保持原来的容量，否则新容量为max(扩容一次的容量，旧容量加上添加的元素个数)。

HashMap底层原理实现

HashMap的底层由数组+链表（红黑树）组成，数组的大小可以在构造方法的时候设置，默认为16，数组中每一个元素就是一个链表，链表到一定数量之后会转化为红黑树，查询效率为logn。

当添加某个元素之后，数组的总的添加长度大于了 数组长度*0.75 ，数组长度扩容为两倍。例如，开始创建HashMap后，数组长度为16，临界值为12，当加入元素后元素个数超过12，数组长度扩容为32，临界值变为24.

在没有红黑树的情况下，添加元素后数组中某个链表的长度超过了8，数组会扩容为两倍。当数组长度扩容为64后，链表就会树化，转换为红黑树（满足了链表树化的两个条件，数组长度达到64，该链表长度达到了8）。

HashSet底层原理实现

HashSet是基于HashMap来实现的，实现了Set接口，同时还实现了序列化和可克隆化。而集合（Set）是不允许重复值的。

所以HashSet是一个没有重复元素的集合，但不保证集合的迭代顺序，所以随着时间元素的顺序可能会改变。

由于HashSet是基于HashMap来实现的，所以允许空值，不是线程安全的。

HashSet是基于HashMap来实现的，区别就是HashMap输入一个键值对，而在HashSet中只输入一个值。

HashSet实现去重是通过判断HashMap的键是否存在，如果存在的话就不添加，如果不存在的话就添加。

引入：红黑树

红黑树是一种自平衡的二叉查找树，是一种高效的查找树。

为什么要出现红黑树？

考虑一下二叉搜索树的缺点：当插入的数据是有序的时候，二叉搜索树的时间复杂度会退化为O(n)。那么为了应对这种极端情况，红黑树应运而生。红黑树是一种接近平衡的二叉树，只是靠着满足红黑结点的五条性质来维持一种接近平衡的结构，进而提升整体的性能，并没有严格地卡定某个平衡因子来维持绝对的平衡。

红黑树满足以下五个特性：

  • 结点是红色或者黑色；

  • 根是黑色；

  • 叶子结点都是黑色（这里的叶子结点指的是最底层的空结点）。

  • 红色结点的子节点都是黑色；红色结点的父节点都是黑色。即，从根节点到叶子节点的所有路径上不能有2个连续的红色结点。

  • 从任一结点到叶子结点的所有路径都包含相同数量的黑色结点。

关于线程池

几个重要的参数

  • corePoolSize: 核心线程数，也是线程池中常驻的线程数，线程池初始化时默认是没有线程的，当任务来临时才开始创建线程去执行任务。

  • maximumPoolSize: 最大线程数，在核心线程数的基础上可能会额外增加一些非核心线程，需要注意的是只有当任务队列填满的时候才会创建多于核心线程数的线程。（线程总数不大于maximumPoolSize）。

  • keepAliveTime：非核心线程的空闲时间超过keepAliveTime就会被自动终止回收掉，注意当corePoolSize=maxPoolSize的时候，keepAliveTime就不起作用了，因为不存在非核心线程。

  • workQueue：用于保存任务的队列，可以为无界，有界，同步移交三种队列类型之一，当池子里的工作线程数大于corePoolSize的时候，这时新进来的任务会被放到队列中。

  • threadFactory：创建线程的工厂类，默认使用Executors.defaultThreadFactory()，也可以使用guava库的ThreadFactoryBuilder来创建。

  • handler：线程池无法继续接受任务（队列已满且线程数达到maximumPoolSize）时候的饱和策略，取值有AbortPolicy，CallerRunsPolicy，DiscardOldestPolicy，DiscardPolicy。

2.CAS机制

原理

CAS(Compare and Swap)，翻译过来为”比较并替换”。CAS机制中使用了三个基本操作数:内存地址V，旧的预期值A，要修改的新值B。

更新一个变量的时候，只有当变量的预期值A和内存地址V当中的实际值相同时，才会将内存地址V对应的值修改为B。

看一个例子：

在内存地址V中，存储着值为10的变量。此时线程1想把变量的值增加1，对线程1来说，旧的预期值A=10，要修改的新值B为11。

在线程1要提交更新之前，另一个线程2抢先一步，把内存地址V中的变量值率先更新成了11。

线程1开始提交更新，首先进行A和地址V的实际值比较，发现A不等于V的实际值，提交失败。

于是线程1重新获取内存地址V的当前值，并重新计算想要修改的值。此时对于线程1来说，A等于11，B等于12。这个重新尝试的过程被称为自旋。

但是呢，自旋并不能代替阻塞，可能多次自旋还是会出现获取不到锁的情况，然后一直占有CPU，白白浪费处理器的时间。所以说，我们需要规定一个自旋时间的限度，默认十次自旋没有成功获得锁，就挂起线程。

3）ABA问题

ABA问题：小灰灰有100块存款，某天他要去银行去20块钱，但由于机器硬件原因，取款操作提交了两次。

于是乎，开启了俩线程：

第一个：获取当前值100，期望更新为80；

第二个：获取当前值100，期望更新为80；

这时候，小灰灰的妈妈也给小灰灰汇款了20块钱。

开启了第三个线程：获取当前值，期望加20.

第一个线程获得了当前值100，第二个线程获得了当前值100，第一个线程把小灰灰的存款更新成了80，第三个线程把小灰灰的存款又更新成了100，然后第二个线程进行CAS操作，发现旧值与地址存的值确实没变化，然后就把小灰灰的存款改成了80.

本来第二个线程应该执行失败的，然后这样就误打误撞重新扣掉了20块。

本质上，cas本身是想改变脏读的情况，但是第三个线程又让脏读回去了，所以会出现问题。

解决问题的方法，就是每次更新加上一个版本号。

原子操作类

原子操作类底层就是用CAS自旋来实现的，synchronized关键字会让没有得到锁资源的线程进入到BLOCKED状态然后来实现进程间的切换，涉及到操作系统内核模式和用户模式的转换，代价非常之高，而原子操作类能很好地通过自旋来解决这一问题。

AQS

AQS全称是AbstractQueuedSynchronizer，意为抽象的队列式同步器，是除了java自带的synchronized关键字之外的锁机制。

AQS的核心思想是，如果被请求的共享资源空闲，则将当前请求资源的线程设置为有效的工作线程，并将共享资源设置为锁定状态，如果被请求的共享资源被占用，那么就需要一套线程阻塞以及被唤醒时锁的分配机制，这个机制AQS是用CLH队列锁实现的，即将暂时获取不到锁的线程加入到队列中。

CLH队列是一个虚拟的双向队列，虚拟的双向队列即不存在队列实例，仅存在节点之间的关联关系。

AQS是将每一条请求共享资源的线程封装成一个CLH锁队列的一个节点，来实现锁的分配。
