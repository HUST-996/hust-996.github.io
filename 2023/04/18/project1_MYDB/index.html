前言

MYDB分为后端和前端，前后端通过socket进行交互。前端（客户端）负责读取用户输入，并发送到后端进行执行，输出返回结果，等待下一次输入；后端则需要解析SQL，如果是合法的SQL，就尝试执行并返回结果。不包括解析器，MYDB的后端包括五个模块：

  • Transaction Manager (TM)

  • Data Manager (DM)

  • Version Manager (VM)

  • Index Manager (IM)

  • Table Manager (TM)

TransactionManager

目录文件

概述

每一个事务都用一个xid来唯一标识，xid占一个字节。规定xid为0的事务是超级事务，当一些操作想在没有申请事务的情况下进行那么可以将操作的XID设置为0。XID为0的事务的状态永远都是commited。

xid这一个字节的空间用来存储该事务的状态，有三种状态：00指active，01指commited，02指aborted（已撤销）。

xid文件的头部留有8字节(位置0到位置7)来记录这个xid文件所管理的事务的个数。因此，一个xid文件可以管理2的64次方个事务。事务xid在文件中的位置为(xid - 1) + 8，字节处，xid为0的事务(超级事务)不需要被记录。

TransactionManager接口

这些接口供其他模块调用，用来创建事务和查询事务的状态。

缓存框架

概述

因为分页管理和数据项都涉及到缓存，所以这里设计一个更为通过的缓存框架，用抽象类来实现。

缓存框架是用引用计数法来实现的，而不是LRU缓存（Least Recent Use）。

设想一个场景：某个时刻缓存满了，缓存驱逐了一个资源，后来上层模块想要把这个资源刷回数据源，这个资源在缓存里却不见了。这个时候就陷入了两难的境地：到底要不要刷回数据源？

由于缓存被驱逐的时间无法确定，我们不知道在这期间数据项有没有被修改。如果被修改的话那回源操作会引发一系列的错误，如果没有被修改的话那这次回源也是一次无效的回源。归根结底还是因为缓存逐出资源的时候上层模块感知不到，而引用计数的策略就不会出现这样的问题，因为引用计数是上层模块主动去释放资源，当引用归零之后缓存就会将这个资源驱逐并回源。

抽象类的设计

cache用HashMap来存储，Long是指该缓存的索引，T是实际缓存的数据类。

references用HashMap来存储，Long是指该缓存的索引，Integer是该缓存的引用次数。

getting用来表示是否有线程正在获取当前缓存。

maxResources表示缓存的最大资源数量，用来限制cache的大小。

count表示当前缓存的元素个数，即cache中索引的数量。

需要锁来实现并发。

这个抽象类还实现了一个get方法，用来根据索引获得想要的资源。判断有没有别的线程在获取，如果在获取那就等待，没有在获取就获得锁来进行资源的获取。如果资源在缓存中，就直接获取；如果不在缓存中，就从数据源处获取，并把数据源里的数据写到缓存里。

还有一个release方法，这个release方法用来形式上地释放资源，其实就是将引用次数-1。真正地释放资源还要再进一步进行判断引用次数是否为0，如果为0的话就真正地将缓存释放，并将缓存写回数据源。

另外还有close方法，就是关闭所有缓存并且将写回数据源。

具体如何从数据源获取数据，如何写回数据，这就得看各个子类地具体实现了。所以getForCache，releaseForCache作为抽象方法进行定义，需要子类去实现。

(Ti,I,A,x),表示事务Ti在A位置插入了一条数据x；

(Ti,U,A,oldx,newx),表示事务Ti将A位置的数据从oldx更新为newx。

为了保证事务之间并发产生的影响，我们做出如下规定：

  • 正在进行的事务，不会读取其他未提交的事务产生的数据。

  • 正在进行的事务，不会修改其他未提交事务修改或产生的数据。

这时候并发情况下日志的恢复策略也就变得简单了：

  • 重做所有崩溃时已完成的事务

  • 撤销所有崩溃时未完成的事务

在恢复后，数据库就会恢复到所有已完成事务结束，所有未完成事务尚未开始的状态。

Version Manager

VM的一个很重要的职责，就是实现调度序列的可串行化。MYDB采用两段锁协议(2PL)来实现，当采用2PL时，如果某个事务i已经对x加锁，且另一个事务也想操作x，但是这个操作与事务i之前的操作相互冲突的话，事务j就会被阻塞。譬如，T1已经因为U1(x)锁定了x，那么T2对x的读或者写操作都会被阻塞，T2必须等待T1释放掉对x的锁。

但是还是不可避免地导致事务之间的相互阻塞，甚至可能导致死锁。

于是，MYDB实现了MVCC，来提高事务的处理效率。

2PL

MVCC

在实现MVCC之前，需要了解事务的隔离级别：

  • 读未提交：可能会导致脏读，不可重复读，幻读

  • 读已提交：可能会导致不可重复读(两次读到的数据不一样)，幻读

  • 可重复读：两次读到的数据一样，但是插入的时候发现已经存在，是幻读

  • 可串行化：最高的隔离级别。

  • 初始：x为0

事务的隔离级别

读已提交

版本的可见性与事务的隔离级别是相关的，为了实现读已提交，MYDB为每个版本维护了两个变量Xmin和Xmax。Xmin指的是创建该记录的事务编号，Xmax指的是删除该记录的事务编号，读已提交就是靠版本的可见性来实现的。

可见条件：

  • 如果记录的Xmin是该事务的id(由该事务创建)，则可见；

  • 如果记录的Xmin是一个已提交的事务id，并且该记录尚未被删除；

  • 如果记录的Xmin是一个已提交的事务id，并且该记录被一个尚未提交的事务删除。

可重复读

读已提交会产生不可重复读的错误，比如事务1在两次读之间有另一个事务开启修改了数据并提交了，事务1虽然读已提交但两次读到的数据是不一样的。

所以我们规定，事务只能读取它开始的时候就已经结束的那些事务提交的数据。

所以，事务需要忽略：

  • 在本事务后开启的事务的数据

  • 在本事务开启时还在活跃的事务的数据

死锁检测

用一个有向图来维护，每次事务出现等待的情况就向图中加入一条边，然后检测是否有环，有环的话就不允许添加，撤销事务抛出异常。
